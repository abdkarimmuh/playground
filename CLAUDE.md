# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Next.js + Tailwind + shadcn/ui playground app. The landing page (`/`) links to five in-browser code playgrounds — `/js` (JavaScript), `/ts` (TypeScript), `/py` (Python), `/rb` (Ruby), `/go` (Go) — all built on one shared Monaco Editor component. Everything executes **client-side only**; there is no backend code execution, by design.

## Before writing any code

**This is not the Next.js you know.** This project pins a Next.js version with breaking changes vs. training-data Next.js — APIs, conventions, and file structure may differ. Read the relevant guide under `node_modules/next/dist/docs/01-app/` before making routing/layout changes, and heed any deprecation notices. Notably, this version has global `PageProps<'/route'>` / `LayoutProps<'/route'>` type helpers (no import needed) generated by `next dev`/`next build`/`next typegen`.

## Commands

```bash
npm run dev         # start dev server (Turbopack)
npm run build        # production build
npm run lint          # eslint (no --fix)
npm run lint:fix     # eslint --fix
npm run format        # prettier --write on **/*.{ts,tsx}
npm run typecheck    # tsc --noEmit
npm run build:wasm   # rebuild public/wasm/yaegi-runner.wasm (requires a local Go toolchain)
```

There is no test runner configured in this repo. Always run `npm run typecheck` and `npm run lint` after changes; run `npm run format` before considering work done (prettier-plugin-tailwindcss reorders class lists, so run it after editing className strings).

`build:wasm` is **not** wired into `predev`/`prebuild` — it requires Go installed locally, and contributors without Go must still be able to run `npm run dev` and get a working `/go` page from the last-committed `public/wasm/yaegi-runner.wasm`. Only re-run it after editing `wasm/yaegi-runner/main.go`.

Adding a shadcn component: `npx shadcn@latest add <name>` (writes into `components/ui/`). The registry style is `base-vega`, which builds on **`@base-ui/react`, not Radix** — components use base-ui's `render` prop for polymorphism (e.g. `<Button render={<Link href="/x" />} />`), not an `asChild` prop. When rendering `Button` as a non-native element, pass `nativeButton={false}` or base-ui logs a console warning.

## Architecture

### Routing
- `app/page.tsx` — landing page, links to the five playgrounds.
- `app/js/page.tsx`, `app/ts/page.tsx`, `app/py/page.tsx`, `app/rb/page.tsx`, `app/go/page.tsx` — thin server components that render `<CodePlayground language="..." .../>` with a language-specific starter snippet and localStorage key. To add another language: add a `LanguageConfig` entry in `lib/playground/languages.ts`, a sandbox builder under `lib/playground/sandbox/`, and one of these page files.
- `app/layout.tsx` — root layout: `ThemeProvider` wraps a `flex min-h-svh flex-col` shell containing `Navbar`, `<main>{children}</main>`, and `Footer`.

### The playground (`components/playground/code-playground.tsx`)
One client component parameterized by `language: PlaygroundLanguage`, reused by all five routes. It knows nothing about any specific language — all per-language behavior is looked up from `lib/playground/languages.ts`'s `LANGUAGES` registry (Monaco language id, an optional `compileToJs` step, and a `buildSandboxSrcDoc` builder). Key mechanics:

- **Editor**: `@monaco-editor/react`'s `Editor`, loaded from jsdelivr CDN by default (no bundler plugin needed — this matters because Turbopack has no equivalent of `monaco-editor-webpack-plugin`). `monaco-editor` is a **devDependency only**, installed solely so TypeScript types resolve correctly for `Monaco`/`OnMount` — it is never imported at runtime (confirm via `@monaco-editor/react`'s dist bundle if this ever seems to regress). Because of this, an `npm install` that drops the optional peer can silently degrade Monaco-related types to `any` without an error — if `getTypeScriptWorker`/`languages.typescript.*` calls stop type-checking correctly, verify `monaco-editor` is still present in `node_modules` and in `package.json`. Monaco's bundled `basic-languages` already tokenize `python`/`ruby`/`go` for syntax highlighting — no extra registration needed. None of the 5 languages get IntelliSense/semantic diagnostics except TypeScript (via the marker mechanism below).
- **Execution**: code runs in a sandboxed `<iframe sandbox="allow-scripts" srcDoc={...}>` — **never add `allow-same-origin`**. For a `srcDoc` iframe that flag makes the sandboxed document same-origin with the parent app itself (there's no independent URL for a `srcdoc` document to derive an origin from), which would let arbitrary user code reach the app's `localStorage`/cookies/DOM. If a runtime's init path trips over the null-origin sandbox, work around it inside the sandbox doc, not by loosening the iframe. Each language's srcDoc is built by `lib/playground/sandbox/{javascript,python,ruby,go}.ts`, all composed from the shared `lib/playground/sandbox/bridge.ts` script (`console.*` overrides + `window.onerror`/`onunhandledrejection`, serializing args to display strings and `postMessage`-ing them to the parent, which appends them to the console panel). The iframe is remounted (`key={runKey}`) on every Run for a clean context.
- **TypeScript compile step**: do **not** use Monaco's `getTypeScriptWorker()` to emit JS — this was tried and reliably raced against the worker's async model-content sync, throwing `Could not find source file: 'inmemory://model/N'`. The current approach (`lib/playground/languages.ts`'s `compileTypeScript`): diagnostics come from `monaco.editor.getModelMarkers()` (main-thread, already computed for the squiggly underlines — no cross-thread query), and JS output comes from the real `typescript` package's `ts.transpileModule()` (a **runtime dependency**, dynamically `import()`-ed only inside the TypeScript code path so `/js` never pays for it). If you're tempted to reach for the Monaco worker API for compilation again, don't — reproduce the marker+transpileModule approach instead.
- **Python** (`lib/playground/sandbox/python.ts`): loads Pyodide from its dedicated jsdelivr path (`cdn.jsdelivr.net/pyodide/v<VERSION>/full/`, **not** the npm-package jsdelivr path), `pyodide.setStdout`/`setStderr({ batched })` pipe straight into the shared bridge's `post()`. Large (~6–9MB) cold download — the sandbox posts a "Loading Python runtime…" log line first so a slow first Run doesn't look hung.
- **Ruby** (`lib/playground/sandbox/ruby.ts`): loads `@ruby/wasm-wasi`'s `DefaultRubyVM` and the CRuby `+stdlib.wasm` binary (~30MB) from jsdelivr. `DefaultRubyVM`'s default `consolePrinter` already writes stdout via `console.log` and stderr via `console.warn` — confirmed by reading the actual package source, not assumed — so the shared bridge's existing `console.*` overrides capture Ruby output with **zero** Ruby-specific stdout plumbing. The only Ruby-specific line is remapping `console.warn` → `post("error", ...)` (in this sandbox it's only ever Ruby's stderr).
- **Go** (`lib/playground/sandbox/go.ts` + `wasm/yaegi-runner/`): there is no practical in-browser Go compiler, so Go code is *interpreted* by [Yaegi](https://github.com/traefik/yaegi), compiled to our own `public/wasm/yaegi-runner.wasm` via `npm run build:wasm` (see below). Uses `GOOS=wasip1 GOARCH=wasm` — **not** the classic `GOOS=js` + `wasm_exec.js` approach, which requires a long-lived instance plus polling for a JS global to appear once `main()`'s `select{}` blocks forever. A `wasip1` module's `_start` runs `main()` to completion in one shot, so `lib/playground/sandbox/go.ts` constructs a **fresh WASI instance per Run** (via `@bjorn3/browser_wasi_shim`, loaded from jsdelivr), matching the existing "remount iframe every Run" philosophy. `wasm/yaegi-runner/main.go` reads source from argv, runs it via `interp.New(interp.Options{Stdout: os.Stdout, Stderr: os.Stderr})` + `i.Use(stdlib.Symbols)` + `i.Eval(src)`; it deliberately never calls `os.Exit` so `wasi.start()` always returns cleanly (exit code 0) and the JS side never needs special exit-code handling — output already streamed via `ConsoleStdout.lineBuffered` callbacks before `main()` returns. Confirmed empirically (native `go run` + Node's `node:wasi`) that: `i.Eval` requires a full `package main`/`func main()` file (bare top-level statements fail to parse), syntax/runtime errors and even explicit `os.Exit`/`panic` calls from *interpreted* user code are all caught by yaegi internally and returned as a normal `error` — none of them crash the host WASM instance. `stdlib.Symbols` intentionally includes the *entire* Go standard library (not a curated subset), which is why the compiled binary is ~40MB (~8MB gzipped) — a deliberate tradeoff (see `wasm/yaegi-runner/`), not an oversight.
- **Persistence**: code is lazily initialized from `localStorage[storageKey]` via a `useState(() => ...)` initializer (safe because Monaco's SSR output doesn't render the code as literal text — it shows a loading placeholder, so there's no hydration mismatch), and saved back on a debounced effect. There's no separate "mounted" gate for this.
- **`useIsClient` (`hooks/use-is-client.ts`)**: use this instead of a `useState(false)` + `useEffect(() => setState(true))` pair for any "has this hydrated on the client yet" flag. The newer `react-hooks/set-state-in-effect` lint rule (part of `eslint-config-next`'s React Compiler rule set) flags synchronous `setState` calls in effect bodies as errors; `useSyncExternalStore`-based `useIsClient` avoids that entirely and is the established pattern here (see `components/navbar.tsx`).

### The Go/Yaegi build (`wasm/yaegi-runner/`)
A standalone Go module (own `go.mod`), *not* part of the Next.js/npm build graph. `npm run build:wasm` runs `go build -C wasm/yaegi-runner -o ../../public/wasm/yaegi-runner.wasm .` with `GOOS=wasip1 GOARCH=wasm`. The output is committed to the repo (`public/wasm/yaegi-runner.wasm`) since most contributors won't have Go installed — only rebuild it after changing `wasm/yaegi-runner/main.go`. Do not try to route Go execution through a third-party server (e.g. the official Go Playground API) instead of Yaegi — that was considered and rejected specifically to keep the app's "everything runs in your browser" property intact.

### CDN dependencies (loaded inside the sandbox iframe, not the parent app)
Pyodide, ruby.wasm's `DefaultRubyVM` + CRuby wasm binary, and `@bjorn3/browser_wasi_shim` are all loaded from jsdelivr **inside each language's sandbox `srcDoc`**, never imported by `code-playground.tsx` or any shared module — this is what keeps `/js` and `/ts` from paying any bundle-size cost for the other three languages. If you ever touch `lib/playground/languages.ts` or the sandbox builders, keep it that way: only reference these runtimes via URL strings inside lazily-invoked builder functions, never via a top-level `import`.

### Styling / theming
- Tailwind v4, config lives in `app/globals.css` (`@theme inline` block), not a `tailwind.config.*` file.
- `ThemeProvider` (`components/theme-provider.tsx`) wraps `next-themes`, attribute `class`, default `system`.
- Sync Monaco's own theme (`vs-dark`/`light`) to `next-themes`'s `resolvedTheme` manually — it's a separate theming system from Tailwind's `dark:` variant.

## Conventions

- No semicolons off — `.prettierrc` has `semi: true`, `trailingComma: "none"`, double quotes, no single-attribute-per-line.
- Import order is enforced by `eslint-plugin-simple-import-sort` (`error`, not just `warn`) — run `lint:fix` rather than hand-ordering imports.
- Version pins in `lib/playground/sandbox/{python,ruby,go}.ts` (Pyodide, `@ruby/wasm-wasi`/`@ruby/3.4-wasm-wasi`, `@bjorn3/browser_wasi_shim`) are hardcoded CDN URL constants, not npm dependencies — these projects ship frequently, so check the current version (`npm view <pkg> dist-tags`/jsdelivr's package page) before bumping rather than assuming the pin is stale or wrong.
- `@typescript-eslint/no-explicit-any` is off project-wide.
- Path alias `@/*` maps to the repo root (see `tsconfig.json` and `components.json` aliases).
