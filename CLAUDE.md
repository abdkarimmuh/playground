# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Next.js + Tailwind + shadcn/ui playground app. The landing page (`/`) links to two in-browser code playgrounds: `/js` (JavaScript) and `/ts` (TypeScript), each built on a shared Monaco Editor component.

## Before writing any code

**This is not the Next.js you know.** This project pins a Next.js version with breaking changes vs. training-data Next.js — APIs, conventions, and file structure may differ. Read the relevant guide under `node_modules/next/dist/docs/01-app/` before making routing/layout changes, and heed any deprecation notices. Notably, this version has global `PageProps<'/route'>` / `LayoutProps<'/route'>` type helpers (no import needed) generated by `next dev`/`next build`/`next typegen`.

## Commands

```bash
npm run dev         # start dev server (Turbopack)
npm run build        # production build
npm run lint          # eslint (no --fix)
npm run lint:fix     # eslint --fix
npm run format        # prettier --write on **/*.{ts,tsx}
npm run typecheck    # tsc --noEmit
```

There is no test runner configured in this repo. Always run `npm run typecheck` and `npm run lint` after changes; run `npm run format` before considering work done (prettier-plugin-tailwindcss reorders class lists, so run it after editing className strings).

Adding a shadcn component: `npx shadcn@latest add <name>` (writes into `components/ui/`). The registry style is `base-vega`, which builds on **`@base-ui/react`, not Radix** — components use base-ui's `render` prop for polymorphism (e.g. `<Button render={<Link href="/x" />} />`), not an `asChild` prop. When rendering `Button` as a non-native element, pass `nativeButton={false}` or base-ui logs a console warning.

## Architecture

### Routing
- `app/page.tsx` — landing page, links to the two playgrounds.
- `app/js/page.tsx`, `app/ts/page.tsx` — thin server components that render `<CodePlayground language="..." .../>` with a language-specific starter snippet and localStorage key.
- `app/layout.tsx` — root layout: `ThemeProvider` wraps a `flex min-h-svh flex-col` shell containing `Navbar`, `<main>{children}</main>`, and `Footer`.

### The playground (`components/playground/code-playground.tsx`)
One client component parameterized by `language: "javascript" | "typescript"`, reused by both `/js` and `/ts`. Key mechanics:

- **Editor**: `@monaco-editor/react`'s `Editor`, loaded from jsdelivr CDN by default (no bundler plugin needed — this matters because Turbopack has no equivalent of `monaco-editor-webpack-plugin`). `monaco-editor` is a **devDependency only**, installed solely so TypeScript types resolve correctly for `Monaco`/`OnMount` — it is never imported at runtime (confirm via `@monaco-editor/react`'s dist bundle if this ever seems to regress). Because of this, an `npm install` that drops the optional peer can silently degrade Monaco-related types to `any` without an error — if `getTypeScriptWorker`/`languages.typescript.*` calls stop type-checking correctly, verify `monaco-editor` is still present in `node_modules` and in `package.json`.
- **Execution**: code runs in a sandboxed `<iframe sandbox="allow-scripts" srcDoc={...}>` (`lib/playground/run-in-sandbox.ts` builds the HTML). The iframe overrides `console.*`, serializes args to strings, and `postMessage`s them to the parent, which appends them to the console panel. The iframe is remounted (`key={runKey}`) on every Run for a clean context. `</script` sequences in user code are escaped before being inlined into the `srcDoc` string.
- **TypeScript compile step**: do **not** use Monaco's `getTypeScriptWorker()` to emit JS — this was tried and reliably raced against the worker's async model-content sync, throwing `Could not find source file: 'inmemory://model/N'`. The current approach: diagnostics come from `monaco.editor.getModelMarkers()` (main-thread, already computed for the squiggly underlines — no cross-thread query), and JS output comes from the real `typescript` package's `ts.transpileModule()` (a **runtime dependency**, dynamically `import()`-ed only inside the TypeScript code path so `/js` never pays for it). If you're tempted to reach for the Monaco worker API for compilation again, don't — reproduce the marker+transpileModule approach instead.
- **Persistence**: code is lazily initialized from `localStorage[storageKey]` via a `useState(() => ...)` initializer (safe because Monaco's SSR output doesn't render the code as literal text — it shows a loading placeholder, so there's no hydration mismatch), and saved back on a debounced effect. There's no separate "mounted" gate for this.
- **`useIsClient` (`hooks/use-is-client.ts`)**: use this instead of a `useState(false)` + `useEffect(() => setState(true))` pair for any "has this hydrated on the client yet" flag. The newer `react-hooks/set-state-in-effect` lint rule (part of `eslint-config-next`'s React Compiler rule set) flags synchronous `setState` calls in effect bodies as errors; `useSyncExternalStore`-based `useIsClient` avoids that entirely and is the established pattern here (see `components/navbar.tsx`).

### Styling / theming
- Tailwind v4, config lives in `app/globals.css` (`@theme inline` block), not a `tailwind.config.*` file.
- `ThemeProvider` (`components/theme-provider.tsx`) wraps `next-themes`, attribute `class`, default `system`.
- Sync Monaco's own theme (`vs-dark`/`light`) to `next-themes`'s `resolvedTheme` manually — it's a separate theming system from Tailwind's `dark:` variant.

## Conventions

- No semicolons off — `.prettierrc` has `semi: true`, `trailingComma: "none"`, double quotes, no single-attribute-per-line.
- Import order is enforced by `eslint-plugin-simple-import-sort` (`error`, not just `warn`) — run `lint:fix` rather than hand-ordering imports.
- `@typescript-eslint/no-explicit-any` is off project-wide.
- Path alias `@/*` maps to the repo root (see `tsconfig.json` and `components.json` aliases).
